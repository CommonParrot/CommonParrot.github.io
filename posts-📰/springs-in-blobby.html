I really wanted to get some of the game feel of the brilliant N++ into Blobby, as it was my biggest inspiration
to begin with. In N++ the physics and dynamic nature of some obstacles are the MVPs, right after the character movement (which I didn't plan to replicate).

One of the more interesting obstacles/platforms were the bouncy cubes.
You see what I mean if you watch this gameplay:

From that came the idea to implement a spring platform for Blobby, which behaved like a real spring would.
For it's gameplay I only needed it to be interactable from above, not from all sides like in N++.
And so I took a look into my Tafelwerk (collection of formulas[1]),
which was collecting dust for too many years, to piece together the physical formulas which would allow
me to get a spring-like behavior.

joint would be a more sensible alternative to this. But in my case much depended on Blobby being a KinematicBody2D instead. While the KinematicBody2D offers more direct control over Blobby's movement, it is mostly incompatible with the built-in physics objects which Godot offers.

The result is a velocity which the spring platform and incoming body share.
For reliability reasons, the velocity of an incoming body is detected via an Area2D,

func body_on_spring() -> void:
    var a_vel = stored_incoming_velocity
    var a_mass = coupled_body.mass
    var b_vel = y_velocity
    var b_mass = mass
    y_velocity +=  (a_mass * a_vel + b_mass * b_vel)/(a_mass + b_mass)
    stored_incoming_velocity = 0
    shock_ready = false

The mass here is the mass of the spring platform itself together with a body, which might have jumped on it.
On the code side, I first register if a body has entered the spring platform, calculating a inelastic shock
with the platform if so. Then I calculate the resulting force like above.
From that I get the acceleration and use the delta time to calculate the current velocity.

func _physics_process(delta: float) -> void:
    var bc = body_contact()
    if !bc:
        shock_ready = true
    if bc && shock_ready:
        body_on_spring()

    var spring_force = spring_k * (self.start_y - self.position.y)
    if coupled_body != null:
        coupled_mass = mass + coupled_body.mass
    else:
        coupled_mass = mass

    var weight_force = coupled_mass * PhysicsConst.gravity
    var result_force = weight_force + spring_force
    y_velocity += (result_force/coupled_mass) * delta

    y_velocity *= friction
    
    # This changes the position of the platform independently from godot physics
    # Blobby then reacts through godot physics with move_and_slide
    self.position.y += y_velocity * delta

I also use a friction variable between 0 and 1, which I apply to the velocity, so that the spring won't oscillate forever.
(the code for body_contact: [3])

3. Energy
...

â€ƒ
When the spring is stretched out, be it by the weight of the platform itself or the combined weight together
with a body (player/creature), it results in some potential energy being stored within the spring.
This is calculated once a body is leaving the Area2D of the spring, giving some of the springs
potential energy back to the body, resulting in a higher jump.

And here is the code for that, including the ratio of the spring platforms weight itself to the body resting on it,
the direction of the resulting velocity and avoiding divisions by , when the mass of the
body resting on the spring gets too low:

func _on_SpringSkin_area_exited(_area: Area2D) -> void:
    var displacement = self.position.y - self.start_y
    var potential_spring_energy = spring_k * 0.5 * pow(displacement, 2)
    var mass_ratio = 1 - mass / coupled_mass
    var transferred_kinetic_energy = potential_spring_energy * mass_ratio
    var kinetic_energy_in_velocity = (
        -sign(displacement)
        * sqrt(
                2 * transferred_kinetic_energy /
                 max(coupled_mass - mass, 0.001) 
        )
    )
    if coupled_body != null:
            # This directly applies a velocity to the coupled body
            # I am mixing physics simulation and kinematic bodies here
            # A game even more centered around physics could benefit from
            # using forces applied to rigid bodies instead 
            coupled_body.velocity.y += kinetic_energy_in_velocity

â€ƒ
The "SpringSkin" here is the Area2D of the spring platform, which triggers interactions when
recognized areas are entering, resting in or leaving. The function itself is connected to and
triggered by a signal of this Area2D instance.

â€ƒ
If I pick up this mechanic for another game, I would probably make it so that the spring accounts
for many bodies resting on it, by expanding on the coupled_body variable.
The velocity is also directly applied to the coupled body from the outside, this could be problematic,
if I add many more such interactions. If my player randomly gets catapulted across a level
it would be hard to trace which mechanic caused the velocity change in the player character.
One idea for the future would be to gather these sorts of physics influences, in a script of it's own to
make them traceable, detect anomalies and apply a single resulting force to the player.
In the small scope of this game it's okay for now.

â€ƒ

Conclusion
...

â€ƒ
The spring is easily one of the most satisfying mechanics in the game right now, if you learn how to use it.
With the tweaks I made to the gravity in the game, the spring constant, the friction and
the weight of the spring platform, a big boost in jump height is more likely than a failed, dampened jump.
But I noticed that one play tester wasn't intuitively waiting for the spring to stretch before jumping again,
loosing a lot of momentum, causing frustration.
I think this indicates that consistent feedback of springs, more independent from players gaming history
and skill, could be more suitable for the widest possible player base, even with the trade-off of mechanical depth.
It might also be mitigated a bit by changing the visuals, to a representation of a real spring and or
adding a color indication, for how much potential energy is in the stretched spring.
I might add that in the future, but for now I enjoy that I could bring over
some of the mechanical depth of N++ to BlobbyðŸ˜„
